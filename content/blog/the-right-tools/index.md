## My Journey in Software Development: Circular Nature of Experience

When an expert does something difficult, they make it look easy. Because of this, the beginner often assumes they can do what the master does. While the gap in their competence might be as wide as the Amazon river, they aren't all that dissimilar. Both operate out of curiosity, ease, and freedom. Without the burden of not knowing enough and knowing enough to know that you don't know enough, the beginner and master are two sides of the same coin.

This is the idea: at some point, experience seems to loop back to the start. In search of simplicity, the master can sometimes be seen as no different from the beginner. 

I've reached a point where I see myself gravitating back to the tools and technology I used when I first started web development. I'm drawn back to simpler times when you could actually see the innards of a website and learn how to build your own website. 

**TL;DR:**
Experience gives you more tools and more knowledge, but choosing the right tool for the job is often overlooked. It matters more than ever when problem solving.

### The Beginnings - Simplicity

My first experience with code was sophomore year of high school. The year was 2013, jQuery was in style, and the holy trinity - HTML, CSS, and JavaScript - felt approachable.

The process was simple:

1. Create a file and call it anything, like `my-great-website.html`
2. Copy and paste a basic skeleton:
   ```html
   <!DOCTYPE html>
   <html>
     <body>
       <h1>My First Heading</h1>
       <p>My first paragraph.</p>
     </body>
   </html>
   ```
3. Write some content: `<h1>Hello world</h1>`
4. Save
5. Open that file in any web browser and voila - you have a website*

📋 **Note:** <a href="/downloads/starter.html" download="my-first-website.html">Download this starter file</a> to create your own website

*Albeit only local on your computer, but still... you created the foundation of the World Wide Web!*

It was during this class where I tasted the potential fruits of coding. If I were to continue learning and building, what could be possible?

Throughout the course, I built fun websites like a planet viewer, a list of the tallest skyscrapers in the world, and a portfolio of the projects we built in class. Creation grew from curiosity and interest. 

All of this was *hand-written* code. Our IDE (Integrated Development Environment - think of this as a fancy Microsoft Word designed for software development) was Dreamweaver, but we were only allowed to use it after proving we could hand-write code in Notepad++.

Looking back, I think this was worth it. Back then, we were forced to actually type and understand what we were doing—especially valuable now, when AI writes so much code.

This even showed up on the final test: what are the hex color codes for common colors? If you are unaware, these values look like `#000444` or `#efefef`. They represent colors, and we were expected to memorize or at least recognize the basic color codes. It was useless, I thought. And I still think it is today.

Anyway, I digress.

With this foundation of code in place, I was no longer asking whether I could code. I was asking myself what to code... And, I was more eager than ever to build.

At this point, the tools I used and the type of code I was writing were as close to the final output as possible.

I would literally write code exactly how it would render in the browser. Today, most professional websites go through pipelines, transformations, and optimizations that greatly change what the developer writes and how it appears in the browser. In short, it is no longer easy to open the hood of a website and see how it was built. This obscures the development process. I often looked under the hood of public websites to learn, and at the time it worked because many websites were simple.

**I classify the beginning of my software journey as the Era of Simplicity.**

### Intermediate Level

I grew from a novice developer with an understanding of how websites worked to a somewhat better developer with the mindset that I could build anything. While wrong, at the time I thought this was true, but looking back there were many caveats.

At this point, I had taken coding more seriously. I was in college and began to learn the ins and outs of web development. This took me down a path of learning new technologies, following coding tutorials, workshops, and classes, and eventually building my own projects. I felt like a capable developer.

I was using tools that augmented how I coded. No longer was the code I wrote the exact code that would be served in the browser. My toolkit expanded to include CSS libraries, databases, JavaScript frameworks, APIs, and more. However, the more I learned, the more complex what I was building became.

While it felt good to use testing suites, automation, and templating for thousands of rendered pages, much of what I used was overkill.

I was not building for massive companies. I was building for myself so I could learn and grow. While this experience helped me, I still was not always using the right tool for the right job.

**This era of perceived experience was the Era of Complexity.**

### Professional Level

My day-to-day job as a software developer has taught me how large companies handle scaling and massive amounts of users. Their level of complexity is through the roof and often justified for their use cases.

At this point, the codebases strayed so far from how they would eventually be viewed in the browser that there were hardly any similarities.

The mental map of how the systems worked grew too complex for any one person to fully understand.

The toolbelt of skills and knowledge outgrew the belt and became a toolbox. This toolbox of libraries, frameworks, and architecture knowledge is great, but it is often too much.

This frustrated me. I didn't want to have to setup my coding environment each time I wanted to build something. Coding was no longer fun, but just a job.

**This is the era of professional software development.**

### Freedom

It has taken me over a decade to find myself coming back to raw HTML and the early web. From 2013 to 2025, I learned how to code, build things with code, but also the monotonous and tedious side of code. Through trial and error, exploring new tech, I've reached a new era: the Era of Freedom.

Today, I choose the least friction to get started and the correct tool for the job. No more boilerplate code copying and pasting. No more over-engineering for a weekend project.

If I want a blog site, I will use a simple static site generator.
If I am building a fun website for myself and friends, I might use React or even just HTML, CSS, and JavaScript.
If I need a scalable cloud solution that can grow to millions of users, I will probably use AWS.

No longer do I feel the need to build with every possible tool.

**This is the Era of Freedom.**

Suited to the title of this blog, here I am, back at the beginning.

With experience, I have gone back to where I was as a beginner, and it feels good.

I do not think expertise grows more complex. In fact, expertise is about learning how to do complex things simply. That is why, when masters are at work, they make what they are doing look easy.
